    Keystone_Token

    关键点

    1. 作为使用API Endpoint的Client ，其数量、类型都是复杂的，因为需要对 Client的身份进行验证
    2. 明文传输密码不完全
    3. Token 作为临时凭证，可以增加认证过程中的安全性
    4. 需要验证 Token 的合法性
    5. API 需要什么东西
      • role
      • Project
      • expire date
    6. Client 需要什么东西
      • Project
      • Catalog

    四种类型

    • UUID tokens
    • Fernet tokens
    • PKI tokens
    • PKIZ tokens

    UUID tokens

    基于 Client 提供的用户名/密码是正确的：

    • keystone:
      • 生成 UUID token
      • 将 UUID token 存储在其后端
      • 将 UUID token 的副本发送回客户端
    • 客户端将缓存 token
    • 然后，UUID token 将随客户端的每个 API 调用一起传递
    • 在收到每次用户的请求后，API Endpoint 会将此 UUID token 发送到 keystone 进行验证
    • keystone 将收到的 UUID Token 与存放在后端的匹配(检查 UUID 字符串，到期日期)
    • keystone 将向 API 端点返回“成功”或“失败”消息

    PKI Token

    每个API Endpoint 都会保留 keystone 的副本：

    • Signing certificate
    • Revocation list
    • CA certificate

    生成CMS Token ，使用如下数据：

    • Service catalog
    • User roles
    • Metadata

    生成代码：

        def _get_token_id(self, token_data):
        try:
        token_json = jsonutils.dumps(token_data, cls=utils.PKIEncoder)
        token_id = str(cms.cms_sign_token(token_json,
                                          CONF.signing.certfile,
                                          CONF.signing.keyfile))
        return token_id

    Token验证和到期

    PKI Token使Openstack API Endpoint能够通过检查Keystone的签名进行令牌有效性和离线验证

    以下三样东西都应该被验证：

    • Token 的签名
    • Token 的有效期
    • Token 是否被删除（撤销）

    被删除的token被强制放入Keystone CA内的撤销列表中。默认情况下，此列表被API Endpoint 每秒从下列url中更新： curl http://[KEYSTONE ip]:3537/V2.0/tokens/revoked 列表的形式是一个简单的json文件：

        {
       "revoked": [
           {
           "expires": "2013-05-27T08:31:37Z",
           "id": "aef56cc3d1c9192b0257fba1a420fc37"
           }
        …
       ]
    }

    CA证书及数字签名

    curl http://[KEYSTONE IP]:35357/v2.0/certificates/signing curl http://[KEYSTONE IP]:35357/v2.0/certificates/ca

    PKIZ Token

    PKIZ 在 PKI 的基础上做了压缩处理，但是压缩的效果极其有限，一般情况下，压缩后的大小为 PKI token 的 90% 左右，所以 PKIZ 不能友好的解决 token size 太大问题。
